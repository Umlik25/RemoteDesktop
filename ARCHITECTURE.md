# App_Remote — архитектура продукта

Кроссплатформенный продукт удалённого доступа к вычислительным ресурсам:
один мощный хост → несколько изолированных пользовательских сессий, включая
локального владельца. Документ описывает целевую архитектуру (v1.0) и честно
отделяет её от того, что реализовано в поставляемом MVP 0.1.

---

## 1. Анализ существующих продуктов (что берём как принцип, не как код)

| Продукт | Полезный принцип | Что применяем |
|---|---|---|
| **Parsec** | Низкая задержка через агрессивное аппаратное кодирование; гостевой доступ с гранулярными правами (мышь/клавиатура/геймпад отдельно); аудит | Права per-устройство-ввода в модели сессии; журнал аудита; приоритет задержки над качеством в игровых профилях |
| **Sunshine + Moonlight** | Self-hosted стриминг без облачной зависимости; NVENC/AMF/QuickSync; согласование кодека клиент↔хост; измерение задержек по стадиям | Конвейер захват→кодирование→сеть→декодирование с метрикой на каждой стадии (виден в оверлее); выбор кодека по возможностям обеих сторон |
| **Tailscale** (mesh-VPN) | Identity-based сеть: устройства находят друг друга через аккаунт; LAN direct / P2P / DERP-relay fallback; deny-by-default ACL; approve нового устройства | Топология «control plane для координации, данные — напрямую»; отображение маршрута; подтверждение устройств; доступ к сессии, а не к сети |
| **Amazon DCV** | Виртуальные сессии как объект первого класса: у сессии владелец, жизненный цикл, права; GPU-sharing на Linux | Session-centric модель данных (см. §9.1); Linux как основная серверная платформа |
| **Citrix / Omnissa Horizon** | Пулы, политики, профили ресурсов, роли, lifecycle-management | Роли owner/admin/user/guest; политики ресурсов per-профиль; отзыв и срок доступа |
| **ASTER Multiseat** | Несколько независимых рабочих мест на одном ПК, включая локального владельца | Локальный владелец = привилегированная сессия, а не «фон» |

Ключевой синтез: **сессия — главный объект системы** (DCV/Citrix), **сеть —
identity-based mesh с прямыми путями** (Tailscale), **стриминг — аппаратный,
с покомпонентным измерением задержки** (Sunshine/Parsec), **владелец —
равноправная локальная сессия с резервом** (ASTER).

## 2. Роли операционных систем

| ОС | Клиент | Хост | Виртуализационный сервер | Комментарий |
|---|---|---|---|---|
| **Linux** | да | **основная целевая** | **да (KVM/QEMU + libvirt)** | Единственная платформа, где реализуемы все режимы: VFIO GPU passthrough, cgroups v2 (жёсткие лимиты CPU/RAM/IO), несколько headless-сессий (Wayland/Xorg virtual, PipeWire), SR-IOV/vGPU при поддержке железа |
| **Windows** | да | да, **с ограничениями** | ограниченно | Ограничения хоста: (1) один интерактивный консольный сеанс на клиентских редакциях — мультисессия только через VM (Hyper-V) или RDS-лицензии на Server; (2) GPU passthrough (DDA) — только Windows Server; GPU-P частично и без официальной поддержки на клиентских SKU; (3) лимиты ресурсов слабее (Job Objects жёстко ограничивают CPU/RAM процессов, но не сеанс целиком). Сильные стороны: лучший захват (DXGI Desktop Duplication), нативные NVENC/AMF/QSV, игры |
| **macOS** | да | **только «лёгкий хост»** | нет | Технически: захват ScreenCaptureKit + VideoToolbox — отличный одиночный хост для работы. Юридически: EULA Apple разрешает виртуализацию macOS только на железе Apple и максимум 2 VM; GPU passthrough отсутствует. Роль: клиент №1, тестовая платформа, single-user-хост для рабочих задач |

**Рекомендация:** хост для мультипользовательских и игровых сценариев —
Linux-машина (или Linux-гипервизор с Windows-гостями для игр). Windows-хост —
для сценария «владелец делится одним потоком» и офисных VM через Hyper-V.
macOS — клиент и одиночный рабочий хост.

## 3. Сеть и подключение (целевая архитектура)

### 3.1 Топология
- **Control plane** (координация): аккаунты, устройства, ACL, обмен ключами,
  сигналинг для NAT traversal. Может быть self-hosted (headscale-подобная
  модель) или облачным. Через него НЕ идёт медиатрафик.
- **Data plane** (медиа и ввод): всегда кратчайший путь:
  1. **LAN Direct** — устройства в одной сети: mDNS/UDP-discovery, трафик
     не покидает LAN, интернет и relay не участвуют. Достаточно логина/пароля
     (MVP уже так работает).
  2. **Internet Direct** — P2P через NAT traversal: ICE-подобная процедура
     (STUN, UDP hole punching, попытка UPnP/PCP), транспорт — WireGuard-туннель
     или QUIC/DTLS напрямую между узлами.
  3. **Relay** — только fallback при симметричных NAT/CGNAT: TURN-подобный
     ретранслятор, шифрование сквозное (relay видит только шифртекст).
- Маршрут всегда показан в UI; смена маршрута — событие в журнале.
- Никаких искусственных лимитов скорости в приложении. Фактическая скорость
  ограничена сетью, RTT, пропускной способностью, энкодером хоста и декодером
  клиента — UI показывает измеренные значения, а не обещания.

### 3.2 Безопасность сети
- Deny-by-default: устройство без approve владельца не получает ничего.
- Ключи устройств: X25519, хранение в OS-хранилище (DPAPI / Keychain /
  libsecret / TPM при наличии).
- Транспорт: WireGuard (Noise) для туннеля или TLS 1.3/DTLS для прямых
  соединений; сквозное шифрование через relay.
- Доступ выдаётся **к сессии**, не к сети хоста: пользователь видит только
  свой стрим-эндпоинт; домашняя сеть хоста недостижима (no subnet routing),
  кроме явно опубликованных ресурсов.

### 3.3 Что в MVP
LAN Direct полностью (UDP-beacon + прямой WebSocket, логин/пароль);
Internet Direct — ручной ввод IP (NAT traversal в дорожной карте);
relay — нет; TLS — нет (только доверенная LAN).

## 4. Стриминг (целевая архитектура)

Конвейер: **захват → препроцессинг → кодирование → пакетизация → сеть →
джиттер-буфер → декодирование → презентация**, каждая стадия измеряется.

| Стадия | Windows | Linux | macOS |
|---|---|---|---|
| Захват | DXGI Desktop Duplication | PipeWire / KMS | ScreenCaptureKit |
| Кодирование | NVENC / AMF / QSV (Media Foundation fallback) | NVENC / VAAPI / AMF | VideoToolbox |
| Декодирование | DXVA2/D3D11VA | VAAPI/NVDEC | VideoToolbox |

- Кодеки: H.264 (базовый, минимальная задержка), HEVC (качество/битрейт, HDR),
  AV1 (лучшая компрессия; только новое железо). Согласование по возможностям
  обеих сторон, ручной выбор — доступен.
- Транспорт видео: RTP-подобный поток по UDP c FEC и NACK (модель
  Moonlight/WebRTC), не TCP: потерянный пакет не должен стопорить кадры.
- Настройки пользователя: разрешение, FPS 30/60/90/120/144 (если дисплей,
  энкодер и сеть позволяют), битрейт/качество, режим (работа / видео / игра /
  низкая задержка), кодек, микрофон, клавиатура/мышь/геймпад, буфер обмена,
  файлы (только по разрешению хоста).
- Звук: Opus 48 кГц; обратный канал микрофона — отдельный Opus-поток.
- Геймпад: виртуальный контроллер на хосте (ViGEm / uinput).

**Текущая реализация:** H.264/NVENC fragmented MP4 по WebSocket/TCP на
совместимом Windows/NVIDIA-хосте; MJPEG-delta остаётся универсальным fallback;
клавиатура+мышь и буфер обмена работают, звук/микрофон/геймпад — следующий этап.

## 5. Ресурсы: что можно ограничить жёстко, а что — только приоритетами

| Ресурс | Жёсткий лимит возможен? | Механизм |
|---|---|---|
| CPU (VM) | **Да** | vCPU count + cgroups cpu.max / Hyper-V limits |
| CPU (сессия-процессы) | Частично | Linux: cgroups — да, жёстко. Windows: Job Objects (rate control) — да для дерева процессов; ниже надёжность для сервисов |
| RAM (VM) | **Да** | balloon/static allocation |
| RAM (процессы) | Да | cgroups memory.max / Job Objects memory limit (риск: OOM-kill приложения пользователя — предупреждаем) |
| Диск (квота) | **Да** | отдельный виртуальный диск / quota / отдельный datаset |
| Диск (IOPS) | Linux — да (io.max), Windows — слабо | |
| GPU (passthrough) | **Да** — вся GPU одной VM | VFIO/DDA |
| GPU (vGPU/SR-IOV/MIG) | **Да**, при поддержке железа и лицензий | NVIDIA vGPU (лицензии!), Intel SR-IOV, AMD MxGPU |
| GPU (time-slicing на общей GPU) | **Нет — только мягкий приоритет** | приоритеты контекстов, ограничение FPS/разрешения стрима как косвенный лимит |
| VRAM на общей GPU | **Нет** (жёстко — только MIG/vGPU) | мониторинг + деградация |
| Сеть | Да | Linux tc/HTB per-сессия; Windows QoS-политики (слабее) |
| Энкодер | Частично | подсчёт слотов NVENC (2–8 сессий у consumer-карт), очередь + приоритет |

Приоритеты (низкий/обычный/высокий/критический) — всегда доступны как мягкий
механизм: nice/cgroup weight/Job priority + порядок деградации стрима.
**UI обязан помечать каждый лимит как «жёсткий» или «приоритет»** — это
уже заложено в модели (MVP реализует мягкие приоритеты + деградацию).

### Автоматический режим
Планировщик каждые N секунд: замер загрузки → сравнение с резервом владельца →
деградация низкоприоритетных сессий по лестнице (см. §10) → восстановление
при спаде. Ручной режим: администратор задаёт лимиты из таблицы выше; система
показывает, какие из них жёсткие на данной ОС.

## 6. GPU и виртуализация: четыре режима

1. **GPU passthrough (VFIO/DDA)** — отдельная GPU на VM. Требования: IOMMU
   (VT-d/AMD-Vi), Linux-хост (или Windows Server для DDA), вторая GPU для
   владельца. Даёт ~нативную игровую производительность. Риск: часть античитов
   детектирует VM (Vanguard, некоторые EAC-конфигурации) — предупреждаем.
2. **vGPU / SR-IOV / MIG** — одна GPU делится аппаратно. NVIDIA vGPU — только
   профессиональные карты + платные лицензии; Intel Arc/Data Center — SR-IOV;
   AMD MxGPU — серверные. На потребительских GeForce официально недоступно —
   продукт обязан говорить это прямо.
3. **Time-slicing общей GPU** — все сессии рендерят на одной GPU хоста
   (как MVP). Подходит для дизайна/видео/1 игровой сессии; производительность
   непредсказуема при конкуренции — только мягкие приоритеты.
4. **Без GPU-виртуализации** — офис/разработка/браузер: программный рендер
   или общая GPU для композитинга, кодирование стрима — главная GPU-нагрузка.

При старте хост определяет доступные режимы (IOMMU, число GPU, драйверы),
показывает их в Capability Report и **не обещает** недоступное. Лицензионные
зависимости (NVIDIA vGPU, Windows Server/RDS CAL, EULA macOS) — отображаются
как требования уровня, а не мелкий шрифт.

## 7. Целевой технологический стек (v1.0)

| Слой | Выбор | Аргументация |
|---|---|---|
| Desktop-приложение | **Rust + Tauri** (UI — TypeScript/Solid или Svelte) | Нативная производительность капчер/кодек-обвязки, малый размер, один код на 3 ОС; системный webview вместо Chromium |
| Медиа-ядро | Rust (crates: `windows`, `nvenc`/FFI, VAAPI) либо обвязка FFmpeg/GStreamer | Прямой доступ к NVENC/AMF/QSV/VideoToolbox; GStreamer — быстрый старт, свой конвейер — минимальная задержка |
| Транспорт медиа | **QUIC (quinn) или собственный RTP/UDP + FEC**; сигналинг — WebSocket/TLS | UDP обязателен для игр; QUIC даёт шифрование+мультиплекс из коробки |
| NAT traversal | ICE-модель: STUN + UDP hole punching + UPnP/PCP; relay-fallback | Открытые стандарты (RFC 8445/5389/8656) |
| Шифрование | WireGuard (туннель устройств) + TLS 1.3 (control plane); ключи в TPM/Keychain/DPAPI | Проверенные примитивы, никакой самодеятельной криптографии |
| Control plane | **Go или Rust**, PostgreSQL, REST+gRPC; self-hosted вариант обязателен | Простое масштабирование, один бинарь для self-hosted |
| Гипервизор | **KVM/QEMU + libvirt** (Linux), Hyper-V (Windows), без хоста-гипервизора на macOS | Открытый, VFIO, зрелый API |
| Оркестрация VM/контейнеров | libvirt XML-шаблоны + cloud-init; лёгкие сессии — systemd-nspawn/cgroups | VM для игр/изоляции, контейнеры для офисных Linux-сессий |
| Мониторинг железа | `sysinfo`/`nvml`/`libdrm`/WMI/IOKit; экспорт Prometheus-метрик | NVML даёт NVENC-слоты, VRAM, температуры |
| Хранение данных | Хост: SQLite (WAL); Control plane: PostgreSQL | Ноль администрирования на хосте |
| Обновления | Tauri updater / собственный канал: подписанные (ed25519) дельта-обновления, staged rollout | |
| Логи/диагностика | `tracing` (Rust) → локальные структурированные логи + опциональный OTLP; кнопка «собрать диагностический архив» | |

**MVP реализован на Python + aiohttp + браузерный UI** — сознательный выбор
для скорости получения работающего продукта; архитектура модулей (control /
capture / session / auth) переносится на Rust-ядро без изменения модели.

## 8. Безопасность и управление доступом (целевая)

- Аккаунт: регистрация/вход, TOTP-2FA, восстановление через одноразовые коды.
- Устройства: список разрешённых, approve нового устройства владельцем,
  отпечаток ключа устройства.
- Роли: **owner** (всё + локальный резерв), **admin** (пользователи, лимиты,
  сессии), **user** (свои сессии), **guest** (ограниченный, срочный доступ).
- Приглашения: одноразовый код/ссылка → конкретному человеку, к конкретной
  сессии, на срок, с профилем ресурсов и прав; мгновенный отзыв. *(в MVP —
  одноразовые коды с ролью/профилем/сроком — работает)*
- Журналы: входы, действия админа, подключения/отключения, смены маршрута,
  деградации. *(в MVP — audit.log + sessions.log, видны в панели)*
- Завершение сессии удалённо, блокировка пользователя, отзыв токенов. *(в MVP работает)*
- Сетевой периметр: доступ только к эндпоинту своей сессии; изоляция VM-сетей
  (NAT без форвардинга в LAN хоста); файлы — только через брокер с разрешением.

## 9. Продуктовая модель

### 9.1 Сессия — главный объект
```
Session {
  id, owner_user, type: local|remote|vm|shared_desktop|gaming_vm,
  status: active|paused|suspended|terminated,
  load_profile: office|dev|design|video|game|competitive|custom,
  resources: {cpu, ram, gpu_mode, vram, disk_quota, net_priority,
              max_resolution, max_fps, max_bitrate, priority},
  permissions: {input, mic, clipboard, files, display, devices[]},
  expires_at, storage_ref, history[]
}
```
MVP хранит подмножество (профиль, приоритет, права ввода/буфера/файлов/дисплея, срок доступа,
историю) — расширение до VM-типов не меняет API.

### 9.2 Владелец — привилегированная сессия
Резерв CPU/RAM/сети всегда; кнопки: стоп новых подключений, приоритеты гостей,
понижение качества гостевых потоков, завершение/пауза сессий, «только работа»,
«игровая сессия владельца» (автоматически поднимает резерв до заданного). *(всё
это есть в MVP-панели; резерв — через governor деградации)*

### 9.3 Профили качества
Офис / Разработка / Дизайн / Видео / Игры / Соревновательные / Пользовательский —
каждый определяет разрешение, FPS, битрейт, кодек, HDR, приоритеты сети/энкодера/
CPU-GPU и поведение при перегрузке. *(в MVP — FPS/качество/масштаб/приоритет)*

## 10. Предсказуемая деградация при перегрузке

Порядок (низкоприоритетные первыми, с уведомлением в оверлее):
1. ↓ битрейт (качество кодирования);
2. ↓ FPS;
3. ↓ разрешение;
4. блокировка запуска новых тяжёлых приложений/сессий;
5. пауза/завершение низкоприоритетной сессии — только после предупреждения.

Продукт никогда не обещает отсутствие потери производительности при любом
числе пользователей. *(лестница 1–3 + предупреждение 5 реализованы в MVP:
`core/host_server.py`, governor)*

## 11. Host Capability Report и Capacity Planner

Перед публикацией хоста: CPU/RAM/GPU/VRAM/диски/сеть, кодеки H.264/HEVC/AV1,
HDR, IOMMU, passthrough/vGPU, число энкодеров, текущая загрузка, качество LAN
и интернета, короткий бенчмарк → четыре уровня:

- **Remote Work Ready** / **Creative Ready** / **Gaming Ready** /
  **Multi-Gaming Ready** — для каждого: доступен ли, почему, ограничения,
  режим GPU, надёжность прогноза.

Capacity Planner: по каждому профилю — рекомендуемое число сессий, доступно
сейчас, узкое место (CPU/RAM/VRAM/GPU/энкодер/сеть/диск), ожидаемое качество,
уверенность, предупреждение о превышении. Это **прогноз, не гарантия** —
формулировка зашита в UI. *(реализовано в MVP: `core/capability.py`)*

## 12. Функции, зависящие от оборудования (никогда не обещаем безусловно)

| Функция | Условие доступности |
|---|---|
| Несколько игровых пользователей на одной GPU | только vGPU/SR-IOV/MIG (профессиональные GPU + лицензии) |
| vGPU | железо+драйверы+лицензии NVIDIA/AMD/Intel |
| GPU passthrough | IOMMU + Linux/KVM (или Windows Server DDA) + свободная GPU |
| HDR | HEVC/AV1 + HDR-дисплей клиента + захват HDR на хосте |
| AV1 | энкодер: RTX 40+/RX 7000+/Arc; декодер на клиенте |
| 120/144 FPS | энкодер+сеть+дисплей клиента; не через relay |
| Высокая производительность через relay | не гарантируется по определению |
| Античиты/DRM в VM и при стриминге | часть игр не работает — показываем предупреждение |

UI показывает фактическую доступность каждой функции на конкретной паре
