# App_Remote — MVP 0.1

Приложение удалённого доступа к ресурсам мощного компьютера: хост публикует
себя в локальной сети, клиенты подключаются под своими учётными записями и
получают стрим рабочего стола с управлением, оверлеем производительности и
профилями качества. Полная архитектура продукта — в [ARCHITECTURE.md](ARCHITECTURE.md).

## Быстрый старт (Windows)

Требуется Python 3.10+ (на этой машине уже установлен 3.12) и зависимости:

```
pip install -r requirements.txt
```

Запуск — двойной клик:

| Файл | Что делает |
|---|---|
| `Запустить.bat` | первый запуск: выбор роли в браузере (хост / клиент) |
| `Запустить как хост.bat` | сразу роль хоста |
| `Запустить как клиент.bat` | сразу роль клиента |
| `Сбросить пароль.bat` | сброс пароля пользователя (если забыли), без остановки сервера |

При первом запуске мастер в браузере спрашивает, будет устройство клиентом,
физическим хостом или VM, предлагает название и автоматически определяет тип
системы. Вводить параметры запуска вручную не требуется. Роль позднее меняется
в настройках панели с автоматическим перезапуском локального сервера.

Необязательные служебные команды для автоматизации:

```
python app.py --reset                          # сбросить выбор роли
python app.py --list-users                     # список пользователей хоста
python app.py --reset-password ИМЯ             # сменить пароль (спросит/сгенерирует)
python app.py --reset-password ИМЯ --password НОВЫЙ
python app.py --role host --node-type physical --host-name "Основной компьютер"
python app.py --role host --node-type vm --host-name "Игровая VM" --parent-host "Основной компьютер"
```

**Забыли пароль?** Три способа, все без остановки сервера и потери данных:
1. **Из UI клиента**: владелец/админ на панели хоста («Пользователи» →
   «Код сброса») создаёт одноразовый код (живёт 1 час); вы вводите его на
   экране подключения клиента («Забыли пароль?») и задаёте новый пароль сами.
2. **Из панели хоста**: «Пользователи» → «Сбросить пароль» — админ задаёт
   пароль напрямую.
3. **С консоли хоста**: `Сбросить пароль.bat` или `--reset-password`.

Во всех случаях меняется пароль одного пользователя; конфиг, другие учётные
записи и активные сессии не затрагиваются, прежние токены входа отзываются.

### Хост
1. В мастере выберите «Предоставлять доступ», укажите тип среды и её название.
   При первом входе в панель (http://127.0.0.1:8532) создайте учётную запись
   **владельца**. При следующих запусках войдите под владельцем или
   администратором. Панель больше не доверяет запросу только потому, что он
   пришёл с этого компьютера.
2. Вкладка «Пользователи» — добавьте пользователей (роль, профиль качества,
   приоритет, права на ввод/буфер обмена/дисплей, макс. FPS), или создайте
   **одноразовый код приглашения** во вкладке «Приглашения».
3. Вкладка «Отчёт о возможностях» — Host Capability Report: железо, кодеки,
   виртуализация, уровни готовности (Remote Work / Creative / Gaming /
   Multi-Gaming) и Capacity Planner с узкими местами.
4. Кнопки владельца: остановить новые подключения, «только работа»,
   «игровая сессия владельца» (поднимает резерв ресурсов), завершение любой
   сессии, блокировка пользователя.
5. **Windows спросит разрешение брандмауэра для python.exe — разрешите**
   (частные сети), иначе клиенты не найдут хост.
6. Вкладка «Настройки» позволяет установить виртуальный дисплей Windows,
   выбрать независимый или физический экран, изменить название и тип среды,
   связать VM с физическим компьютером и переключить приложение в режим клиента.

### Клиент
1. Панель (http://127.0.0.1:8600) сама находит физические хосты и VM в LAN
   (UDP broadcast, порт 8533), показывает их отдельными типами и позволяет
   отфильтровать список; трафик идёт напрямую к выбранной среде по LAN
   (LAN Direct), без интернета, relay и двойной передачи через гипервизор.
2. Кнопка «Добавить среду» сохраняет хост или VM по IP:порт. Сохранённые среды
   не исчезают после перезапуска и доступны для редактирования и удаления.
3. «Подключиться» → логин/пароль (или код приглашения) → выбор профиля
   качества, FPS, качества и масштаба → открывается окно сессии.
4. В сессии: кнопка **📊 Оверлей** (FPS, frame time, ping, jitter, битрейт,
   кодек, разрешение, маршрут, загрузка CPU/GPU/RAM/VRAM хоста, предупреждения
   о деградации), смена профиля/FPS/качества на лету, буфер обмена и личные
   файлы на хосте (если разрешены), полноэкранный режим. Зелёный индикатор
   рядом с настройками показывает подтверждённые хостом разрешение, FPS и Q;
   жёлтый — ограничение учётной записи, рабочего режима, нагрузки или сети.
   - **Разрешение потока** выбирается списком реальных разрешений (родное /
     1440p / 1080p / 900p / 720p — по возможностям хоста). По умолчанию «Авто»:
     для хостов выше 1200p стартует с 1080p ради плавности.
   - **Экран ноутбука / Текущее окно** показывает реально определённый браузером
     размер в физических пикселях и отдельно меняет системное разрешение Windows
     на ближайший поддерживаемый монитором режим с нужными пропорциями.
     Это право выдаёт владелец; режим общий для всех и автоматически
     восстанавливается при отключении. Масштаб потока при этом остаётся
     независимой настройкой качества и битрейта.
   - Верхняя панель сворачивается кнопкой **⌃**, появляется у верхнего края или
     по кнопке **⌄** и больше не раскрывается от движения удалённой мыши.
   - **🔒 Захват мыши** (Pointer Lock): курсор не выходит за окно и не задевает
     меню/док macOS при движении к краям экрана. **Esc** освобождает курсор.
     На Mac включён по умолчанию — клик по картинке захватывает мышь; после
     входа в полноэкранный режим захват возвращается автоматически.
   - **Режим: работа** использует привычное ускорение мыши и печатает символы
     **как есть** (Unicode) —
     точка, запятая, кириллица работают при любой раскладке клиента и хоста.
     Язык переключаете у себя (на Mac), раскладку Windows-хоста трогать не
     нужно; кнопка **🌐** шлёт хосту Win+Space, если раскладка нужна приложению.
     **Режим: игра** передаёт движение мыши 1:1 без ускорения и клавиши как
     физические кнопки (WASD с удержанием); включается автоматически для
     профилей Игры/Соревновательные.
   - **⌘→Ctrl**: на Mac клавиша ⌘ (Cmd) отправляется как Ctrl (⌘C = копировать
     на Windows и т.п.). Кнопкой можно отключить. Часть системных сочетаний Mac
     (⌘W, ⌘Q, ⌘Tab, ⌘Space) перехватывает сама macOS — до окна они не доходят.
   - Курсор хоста показывается **своей формой** (стрелка/текст/рука/resize) и
     двигается мгновенно.

### Подключение к виртуальной машине

1. Настройте VM с сетевым мостом (bridged network), чтобы она получила
   собственный IPv4-адрес в той же LAN. При NAT используйте ручное добавление
   IP/порта и настройте проброс TCP 8532; UDP-обнаружение через NAT обычно не проходит.
2. Установите приложение и зависимости **внутрь гостевой ОС**. В мастере
   выберите «Предоставлять доступ» → «Виртуальная машина», задайте название VM
   и при необходимости название физического компьютера.
3. Создайте владельца и пользователей в панели VM. Её учётные записи, файлы
   и рабочий стол независимы от установки приложения на физическом хосте.
4. На физическом компьютере в том же мастере выберите «Физический компьютер».
   Клиент показывает обе среды и подключается к каждой напрямую.

Тип среды определяется автоматически по данным гостевой системы, но его можно
явно изменить в панели хоста: «Настройки → Тип среды». App_Remote не создаёт VM
и не назначает ей GPU: это делает гипервизор. Агент использует ту GPU и тот
виртуальный дисплей, которые гостевая ОС уже видит.

### Если хост не находится автоматически
- Хост и клиент должны быть в **одной локальной сети** (один Wi-Fi/роутер).
  Обнаружение — UDP broadcast, между разными сетями оно не проходит.
- **VPN-адаптеры** (Radmin VPN, Hamachi — адреса `25.x`/`26.x`) раньше
  перехватывали broadcast; теперь хост рассылает beacon направленно по каждой
  реальной подсети и исключает VPN-диапазоны. При старте хост печатает адреса
  рассылки: `LAN-обнаружение: рассылка на ['192.168.0.255', ...]`.
- **Брандмауэр** на хосте (Windows) и клиенте (Windows/macOS) должен пропускать
  python на порт 8533/UDP. На macOS при первом запуске система спросит доступ.
- Гостевые/изолирующие Wi-Fi-сети и «AP isolation» на роутере блокируют
  broadcast между устройствами — тогда добавьте хост **вручную по IP:порт**
  (IP хоста виден в его консоли и в панели).

## Что реализовано в MVP

- Роли хост/клиент с выбором при первом запуске.
- Обнаружение физических хостов и VM в LAN + прямое подключение (LAN Direct),
  фильтрация по типу среды и ручное добавление по IP.
- Учётные записи (PBKDF2), роли owner/admin/user/guest, блокировка, отзыв,
  срок доступа, одноразовые приглашения, журнал аудита и история сессий.
- Защищённая панель хоста: обязательный вход owner/admin, короткоживущий токен
  только на время вкладки, ограничение попыток входа и browser-origin. Новые
  пароли — минимум 8 символов; существующие учётные записи продолжают работать.
- Истёкший или отозванный доступ завершает активный поток; изменение прав на
  ввод и буфер обмена применяется к уже открытой сессии.
- Одноразовый код приглашения или сброса помечается использованным только после
  успешного создания пользователя/смены пароля.
- Передача файлов по разрешению владельца: отдельное хранилище и дисковая квота
  для каждого пользователя, загрузка, скачивание и удаление из окна сессии.
  Имена файлов проверяются, каталоги пользователей изолированы друг от друга.
- Стриминг экрана автоматически выбирает **H.264/NVENC** на совместимом
  Windows-хосте и браузере. FFmpeg захватывает Desktop Duplication и передаёт
  D3D11-кадры в NVENC без копирования полного 4K BGRA-кадра через Python;
  fragmented MP4 идёт по прямому WebSocket и аппаратно декодируется браузером.
  B-кадры/lookahead отключены, MP4 фрагментируется по кадрам, клиент держится у
  живого края буфера. Это основной путь для родного 1440p/4K и 60+ FPS.
- Совместимый запасной поток — **дельта-кадры MJPEG**: передаются
  только изменённые области экрана (бинарный формат v2: 24-байтовый заголовок
  с full/delta, координатами, номером и временем захвата + JPEG области).
  Полный кадр — при подключении, смене параметров, по запросу клиента
  (`refresh`) и редко как контрольный
  после дельт. На типичной работе это снижает битрейт в 5–40 раз (замер:
  ~37 → ~7 Мбит/с при 60 FPS), что напрямую убирает задержку по Wi-Fi.
  Захват через DXGI Desktop Duplication (60+ FPS, fallback — GDI), поиск
  изменённой области — numpy, кодирование — OpenCV/libjpeg-turbo (fallback —
  Pillow), отправка событийная (без потерь FPS на таймере Windows).
  Курсор в кадр НЕ впечатывается: форма и позиция идут отдельным каналом
  (~30 Гц), клиент рисует курсор сам прямо в canvas (не DOM-слой — не может
  пропасть из-за fullscreen/pointer-lock причуд). Выбор разрешения потока
  списком реальных разрешений. JPEG не сжимается повторно на WebSocket,
  устаревший результат кодирования отбрасывается, клиент держит не больше двух
  зависимых кадров и подтверждает фактическое декодирование; адаптация
  использует эту очередь и измеренную задержку, а не только запись в сокет.
- Ввод: мышь (рабочее ускорение по скорости движения; игровой Pointer Lock 1:1
  без ускорения; изоляция от краёв
  экрана клиента, боковые кнопки, горизонтальная прокрутка), клавиатура в двух
  режимах — «текст» (Unicode-набор, не зависит от раскладок) и «игра»
  (физические клавиши, раскладко-независимые шорткаты Ctrl+C и т.п. через
  виртуальные коды), раскладка Mac→Windows (⌘→Ctrl), кнопка смены языка хоста
  (Win+Space), буфер обмена по разрешению. Клики и клавиши уходят немедленно,
  движения мыши коалесцируются (~8 мс).
- Автоадаптация к каналу и кодировщику: если сеть, захват или JPEG не успевают,
  качество снижается ступенями (масштаб/качество → FPS) и восстанавливается
  само. Адаптация учитывает
  скорость физического LAN-порта и не принимает VPN/loopback за гигабитную сеть.
  При линке 100 Мбит/с интерфейс показывает явное предупреждение. Переключатель
  **Авто / Фикс.** позволяет отключить автоматическую адаптацию и удерживать выбранные
  значения; аварийный governor хоста и административные лимиты сохраняются.
- Профили качества (офис/разработка/дизайн/видео/игры/соревновательные),
  смена FPS/качества/точного масштаба на лету с подтверждением хоста.
- Host Capability Report + Capacity Planner (прогноз с уровнем достоверности).
- Резерв владельца и предсказуемая деградация при перегрузке:
  битрейт → FPS → разрешение → предупреждение о паузе.
- Оверлей производительности в каждой сессии.

## Честные ограничения MVP

- **Один рабочий стол на узел.** Агент на физическом хосте показывает общий
  рабочий стол хоста, агент внутри VM — общий рабочий стол этой VM. Отдельная
  VM уже подключается как независимая среда, но автоматическое создание VM,
  запуск отдельной VM на каждого пользователя и управление GPU passthrough
  остаются этапами 0.5–0.6 (Linux/KVM, Hyper-V).
- **Независимый экран требует одноразовой установки IDD.** На Windows-хосте
  откройте «Настройки» → «Экран трансляции» и установите виртуальный дисплей.
  После перезапуска режим «Авто» выбирает его первым: DXGI/MSS/NVENC, разрешение
  и мышь привязываются к виртуальному output, а подключённый физический монитор
  не используется и не переключается. Без IDD приложение безопасно откатывается
  к физическому экрану.
- **Один выбранный экран остаётся общим для сессий одного узла.** Независимый
  виртуальный экран отделён от локального монитора владельца, но отдельный
  рабочий стол на каждого удалённого пользователя по-прежнему требует VM или
  отдельной Windows-сессии.
- **H.264/NVENC пока только на Windows + NVIDIA.** Если FFmpeg, NVENC,
  `ddagrab` или H.264 MSE недоступны, приложение автоматически возвращается к
  MJPEG и показывает причину. AMF/QSV/VideoToolbox и UDP-медиатранспорт ещё не
  реализованы; TCP остаётся чувствительным к потере пакетов вне стабильной LAN.
- **Без TLS.** Использовать в доверенной локальной сети. Шифрование
  транспорта — этап 0.2/0.4.
- **Без NAT traversal/relay.** Интернет-подключение — только вручную
  (проброс порта/VPN). P2P и relay-fallback — этап 0.4.
- Звук, микрофон и геймпад — этап 0.2.
- Жёсткие лимиты CPU/RAM/GPU на Windows-хосте не применяются — работают
  мягкие приоритеты и деградация потоков (см. ARCHITECTURE.md §5).
- Фактическая скорость и качество зависят от сети, задержки, CPU/GPU
  кодирования и клиента — приложение показывает измеренные значения
  и не обещает «безлимитных» характеристик.

## Порты

| Порт | Назначение |
|---|---|
| 8532/TCP | HTTP/WS хоста (панель, API, стрим) |
| 8533/UDP | LAN-обнаружение (broadcast) |
| 8600/TCP | локальная панель клиента (только 127.0.0.1) |
| 8599/TCP | страница выбора роли при первом запуске |

## Структура

```
app.py                — точка входа (выбор роли, запуск серверов)
core/state.py         — конфиг, аудит, история сессий
core/auth.py          — пользователи, роли, токены, приглашения
core/hwinfo.py        — определение железа, загрузки, бенчмарк
core/capability.py    — Capability Report, Capacity Planner, профили
core/host_server.py   — сервер хоста: API, стриминг, governor, beacon
core/client_app.py    — клиент: панель, LAN-discovery
web/                  — интерфейс (setup / host / client / viewer)
data/                 — создаётся при работе: config, users, журналы
```
